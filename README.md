# Growth Experiment Framework

**增长实验框架：把“要增长”拆成可以验证的假设和动作**

做新品、新业务或新渠道增长时，最常见的问题不是没有动作，而是动作太散。

今天想测素材，明天想换受众，后天又觉得落地页不行。每个动作看起来都合理，但如果背后没有清楚的假设，最后复盘时就很容易只剩一句话：

> 这次效果不好，下次再试试别的。

这其实很可惜。因为增长实验最有价值的地方，不只是某一次结果好不好，而是每一次测试之后，团队是否更清楚：

- 谁更可能需要这个产品？
- 用户在什么场景下会产生需求？
- 哪个痛点更容易触发行动？
- 哪种内容更能建立信任？
- 哪个环节阻碍了转化？
- 下一轮应该放大、复测、暂停，还是补数据？

所以我做这套框架时，想解决的不是“写一份漂亮的实验计划”，而是把模糊的增长目标拆成一组可以被验证、被复盘、被继续推进的动作。

## 我为什么要做这套框架

很多增长问题一开始都很模糊。

比如：

- 新品上线后不知道先投哪个渠道；
- 素材跑了几条，但不知道为什么有的好、有的差；
- 落地页改了几版，但不知道到底影响了哪个环节；
- 广告花费有了，但转化不稳定；
- 团队复盘时只讨论结果，没有沉淀原因。

这些问题如果直接进入执行，很容易变成“多做一点、多投一点、多试一点”。但增长不是靠动作堆出来的，尤其在预算有限、样本有限、历史数据不足的时候，更需要先把问题拆清楚。

我更倾向于先问：

> 这次增长动作到底想验证什么？

只要这个问题没有回答清楚，后面的素材、渠道、落地页、优惠、预算和复盘都容易失焦。

---

## 我的核心判断

一个增长实验不应该只写“测试 Meta 广告”或“测试一批素材”。这类描述只能说明动作，不能说明判断逻辑。

我更希望每个实验都能说清楚这 6 件事：

| 问题 | 我真正想确认什么 |
|---|---|
| 业务问题是什么 | 当前要解决的是获客、转化、复购、留存还是规模化 |
| 目标用户是谁 | 不是泛人群，而是具体场景下有需求的人 |
| 用户为什么会行动 | 痛点、动机、触发场景和购买理由是什么 |
| 我要测试什么 | 渠道、素材、内容角度、落地页、权益或转化路径 |
| 用什么指标判断 | CTR、ATC、CVR、CPA、ROAS、GMV 等指标各自回答什么问题 |
| 结果出来后怎么办 | 成功后放大，失败后优化、复测、暂停还是补数据 |

如果这 6 件事没有写清楚，实验就很难复盘。因为你只能知道“结果如何”，却很难知道“为什么是这个结果”。

## 我会怎么拆一个增长问题

### 第一步：先定义业务问题

我不会一开始就问“投什么渠道”，而是先判断当前业务卡在哪里。

| 业务阶段 | 典型问题 | 更适合先验证什么 |
|---|---|---|
| 新品冷启动 | 不知道有没有需求 | 用户痛点、场景、内容角度 |
| 早期验证 | 有点击但转化不稳 | 信任、价格、落地页、购买阻力 |
| 渠道测试 | 不知道哪个渠道更有效 | 渠道意图、流量质量、转化成本 |
| 素材测试 | 不知道什么内容能带来行动 | Hook、证言、卖点、CTA |
| 放量阶段 | 小预算有效，大预算跑不动 | 素材规模、受众扩展、频次、边际成本 |

这一步的重点是先把问题范围缩小。否则“增长不好”这个问题太大，任何动作都像是可能的解决方案。

### 第二步：把用户和场景说清楚

新品增长不是先找流量，而是先找“谁在什么情况下真的需要它”。

我会把用户假设拆成这几个字段：

| 用户人群 | 核心痛点 | 触发场景 | 购买动机 | 购买阻力 | 内容角度 |
|---|---|---|---|---|---|
| 示例：高频使用某类产品的人 | 示例：当前方案效率低或体验差 | 示例：遇到具体任务或问题时 | 示例：希望更省时、更稳定、更有安全感 | 示例：担心价格、效果、售后或学习成本 | 示例：痛点前置、结果展示、真实证言 |

如果这张表写不出来，说明现在还不适合直接放大投放。因为你还不知道该对谁说、说什么、为什么对方会信。

### 第三步：把动作写成可验证假设

我会尽量避免只写：

```text
测试素材
```

更好的写法是：

```text
如果目标用户在某个场景下对某个痛点高度敏感，
那么围绕这个痛点的内容应该在点击率、互动率或加购率上高于普通卖点内容。
```

这个写法的好处是，实验结束后你能判断的是“痛点内容是否成立”，而不只是“这几条素材好不好”。

### 第四步：给指标分层

我不会只看最终 ROAS。尤其在新品早期，样本少、预算小、学习期波动大，只看最终转化很容易误判。

| 层级 | 常见指标 | 主要回答什么 |
|---|---|---|
| 前链路 | CTR / Hook Rate / 互动率 | 内容是否能吸引用户 |
| 中链路 | LPV / ATC / IC | 用户是否产生兴趣 |
| 后链路 | CVR / CPA / ROAS | 是否具备转化效率 |
| 业务层 | GMV / 毛利 / 回本周期 | 是否值得继续放大 |
| 内容层 | 素材生命周期 / 复用率 | 是否能沉淀内容资产 |

指标不是越多越好。一个实验最好有 1 到 2 个核心指标，再配少量辅助指标，否则复盘会变成到处找理由。

### 第五步：提前写清下一步动作

很多实验的问题不是没有结果，而是结果出来后不知道怎么办。

所以我会在实验开始前就写清楚：

- 达到什么标准就放大；
- 没达到但有局部信号时怎么复测；
- 哪些情况说明要暂停；
- 哪些情况说明不是素材问题，而是落地页、价格、受众或信任问题；
- 数据不足时还需要补什么。

这样复盘时就不会只停留在“好 / 不好”，而是能自然进入下一轮动作。

## 快速使用

如果只是想马上开始拆一个增长实验，可以先不用任何复杂模板，直接用下面这两张表。

### 1. 用户与场景假设表

| 用户人群 | 核心痛点 | 触发场景 | 购买动机 | 购买阻力 | 内容角度 | 优先级 | 备注 |
|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |  |

这张表先解决“对谁说、在什么场景下说、说什么更可能触发行动”的问题。

### 2. 增长实验计划表

| 实验编号 | 业务阶段 | 业务问题 | 目标用户 | 核心痛点 | 触发场景 | 增长假设 | 测试动作 | 渠道 | 内容角度 | 核心指标 | 成功标准 | 预算/资源 | 实验周期 | 下一步动作 | 状态 | 复盘结论 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| EXP-001 |  |  |  |  |  | 如果...那么... |  |  |  |  |  |  |  |  | 待开始 |  |

这张表的重点不是填得很满，而是让每个实验都有一个明确假设、一个测试动作、一个核心指标和一个下一步判断。

如果要检查实验计划是否漏填关键字段，也可以使用仓库里的校验脚本：

```bash
python3 scripts/validate_experiment_plan.py examples/example_experiment_plan.csv output/invalid_rows.csv
```

脚本会检查必填字段是否为空、实验编号是否重复，并把有问题的行输出到 `invalid_rows.csv`。

## 常见复盘判断

实验结束后，我会尽量避免只说“表现好”或“表现差”。更有价值的是把结果和可能原因对应起来。

| 结果表现 | 可能原因 | 下一步动作 |
|---|---|---|
| CTR 高，CVR 低 | 内容吸引人，但信任、价格或落地页不足 | 优化落地页、补充证言、调整权益 |
| CTR 低，CVR 高 | 内容吸引力弱，但进入后的用户意图强 | 优化 Hook，扩展内容角度 |
| ATC 高，购买低 | 用户有兴趣，但购买阻力强 | 优化价格、物流、支付、信任信息 |
| ROAS 高但跑不动 | 素材或受众规模有限 | 扩展素材矩阵，测试更多场景 |
| CPC 高但 CVR 好 | 流量贵但意图强 | 评估毛利和规模，不急着否定 |
| 花费少、数据波动大 | 样本不足 | 延长观察或降低结论强度 |

复盘不是为了证明某个人判断对错，而是为了让下一轮测试更清楚。

## 适用场景

- 新品冷启动；
- 新业务增长验证；
- 内容增长实验；
- 用户获取策略设计；
- 广告素材测试；
- 营销活动复盘；
- 渠道策略拆解；
- 落地页承接与转化路径优化；
- 增长团队内部协作。

## 我希望作品集传达的能力

这个项目想表达的不是“我会写实验计划表”，而是：

1. 我能把模糊的增长目标拆成清楚的业务问题。
2. 我能从用户、场景、痛点和购买阻力出发设计实验。
3. 我知道增长动作必须对应假设，否则结果很难复盘。
4. 我能把内容、渠道、落地页、指标和下一步动作串成一条验证链路。
5. 我能用表格和脚本把方法沉淀成可复用工作流，而不是只停留在经验判断。
